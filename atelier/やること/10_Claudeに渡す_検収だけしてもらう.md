# 10. 検収だけしてもらう(使い回し)

> **■使い方(3行)**
> ① CodexのPRをマージしたら、**新しい**Claude Codeセッション(biz001)を開く
> ② 「`atelier/やること/10_Claudeに渡す_検収だけしてもらう.md` を読んで、直近にマージされた変更を検収して。対象のお願い書は やること/◯番」と言う(◯番を実際の番号に)
> ③ 検収報告(合格 or 修正済み)とスクリーンショットを確認する

なぜこれを分けるか: 実装は量産級(Codex)に任せ、最高性能のAI(Claude/Fable5級)は**検収という短い仕事だけ**に使う(憲法第7条)。Codexの環境はブラウザ検証ができないことがあるため、実機での動作確認は毎回この検収で担保する(実績: Pivot Bench磨き込みの検収で実バグ2件を発見・修正した)。

## あなた(検収者)の仕事

**実装のやり直しはしない。検証と、見つかった不具合の最小修正だけを行う。**

1. `atelier/憲法.md`(特に第8条)と、指定された「対象のお願い書」(やること/◯番)を読む
2. mainを取り込んだ作業ブランチを作る(`git fetch origin main && git checkout -B <指定ブランチ名> origin/main`)
3. **机上レビュー**: マージされた差分を読み、疑わしい箇所をリストアップしてから実機で確定させる(推測で直さない)
4. **実機検証**:
   - ビルド一発再現(`npm run build` 等)と、リポジトリにあるテストの実行
   - Playwright(このリポジトリの過去セッションの作法: filechooser・toast待ち・ダウンロード検査・`page.on('request')`でのネットワーク0件監視・`locale: 'ja-JP'`)で、対象お願い書の**合格チェック項目を1つずつ**検証するスクリプトを書いて実行
   - 既存機能の回帰確認(壊れていないか)
5. **不具合が出たら**: 原因を実機で確定→最小修正→全テスト再実行。修正はやり過ぎない(スコープは対象お願い書の範囲のみ)
6. **記録と取り込み**: `atelier/全体計画.md` の記録表に検収結果を1行(発見した不具合と修正内容を含む)、対象お願い書のstatusを完了に。コミット→push→mainへのPR作成→マージ
7. **報告**: 合否、発見・修正した不具合(事実→原因→修正)、テスト結果の数字、スクリーンショットを添えて持ち主へ報告

## 合格チェック(検収作業そのものの)

- [ ] 対象お願い書の合格チェック全項目に実機の証拠がある(「Codexがテストしたと言っている」は証拠ではない)
- [ ] 実行時ネットワーク0件・localStorage不使用の維持を確認(Webツールの場合)
- [ ] スクリーンショットで格好良さを目視確認(憲法第4条)
- [ ] 記録表更新とmainへの取り込みまで完了
