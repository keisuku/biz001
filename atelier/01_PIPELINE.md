# 01 PIPELINE — テーラーメイド手順書

「インサイト確定 → 取捨選択 → 最高峰事例確認 → 紹介 → 取捨選択 → カスタマイズ」を、誰(どのモデル)がやっても同じ品質で回せるように手順化する。1つの欲望 = 1つのインサイトファイル。途中で止まっても、ファイルを見れば次の段階が分かる。

```
[兆し] → ① インサイト確定 → ② 需要の取捨選択 → ③ 最高峰事例確認
                                                      ↓
       ⑥ カスタマイズ ← ⑤ 実装の取捨選択 ← ④ 紹介 (既存で足りればここで終了=最良)
```

## 第0段階: 兆し(いつでも・数秒)

「〜が欲しい気がする」「〜が面倒」と思った瞬間に `templates/insight.md` を複製して1行書く。整理不要。書く場所は vault の Inbox(または本リポジトリ `atelier/insights/`)。

## ① インサイト確定 (対話推奨: Fable5級)

チェックリスト — 全部埋まるまで次に進まない:
- [ ] **動詞**: 何をする道具か(例: 「集計して並べて比較する」)
- [ ] **頻度**: 週何回使うか(週1未満なら作らない選択も検討)
- [ ] **場面**: いつ・どこで・何のデータで
- [ ] **現状**: 今は何でどうやっていて、何分かかっているか
- [ ] **成功の絵**: 完成した道具を使っている自分の情景を3行で

## ② 需要の取捨選択

機能候補を **must / nice / 不要** に分ける。判断基準:
- 週1回以上使う操作か → must
- 「あったら便利そう」だけで場面が言えない → **不要**(ここで切る勇気が品質)
- 格好良さ・操作性の要件もここで must に明記する(第4条)

## ③ 最高峰事例確認 (委譲可: リサーチ系モデル)

世界単位で調べ尽くす。`templates/research.md` を使い `research/Rxx_*.md` に残す。
- GitHub(スター順・直近更新)、商用製品、有名アプリ、HN/Reddit の定番スレ
- 各候補につき: 何が最高峰か / 盗むべき機能・UX / 捨てる部分 / ライセンス
- 調査の指示書化は [briefs/B02_リサーチ網.md](briefs/B02_リサーチ網.md)

## ④ 紹介

調査結果から「そのまま使えば終わりか?」を判定する。
- **足りる** → 導入して終了。実装ゼロが最良の結果(第2条)
- **足りない** → 何が足りないか(差分)を1段落で言語化。差分こそが作るものの定義

## ⑤ 実装の取捨選択

**引き算の設計**: 最高峰事例のどの機能・UXを盗み、どれを捨てるか列挙する。
ベース(土台にするOSS/既存資産)を1つ決める。自作するのは差分のみ。

## ⑥ カスタマイズ

- `templates/brief.md` で指示書を書く(自己完結: 背景・参照最高峰・仕様・検収基準)
- [03_MODEL_ROLES](03_MODEL_ROLES.md) でモデルに割当て
- 実装 → **検収(第8条: 動作証跡必須)** → [02_MASTERPLAN](02_MASTERPLAN.md) に反映
